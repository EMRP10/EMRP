Welcome

This website is for the Extract Method Refactoring Datasets Platform (EMRP).


EMRP is an open source dataset of extract method refactoring intended for supporting software engineering researchers and practitioners. For each case in EMRP, we provide a detailed study and utilization of dataset for extract method refactoring techniques. EMRP currently provides datasets studies for 7 refactoring methods.


Here you can find useful information, such as how to download the dataset, find support and understand the dataset.


About

Extract Method Refactoring Platform, a.k.a. EMRP, is an open-source repository for Extraction Method refactoring methodologies. This platform collects and manages the relevant papers, datasets, and tools for utilization by researchers. EMRP was first designed by Addl Tariq as part of his MS Capstone at Rochester Institute of Technology, Rochester. Dr. Mohamed Wiem Mkaouer and Dr. Eman Abdullah AlOmar have also contributed to the first incarnation of EMRP.


EMRP collects information from open-source publications and reviews and provides information about the implemented techniques, dataset type and quality, and tools for replication. By leveraging the data contained in EMRP, software engineering researchers and practitioners can review the historical research on the Extract Method topic, and can utilize the existing studies (techniques, datasets, and tools) to improve or create new empirical studies to assess Extract Method Refactoring. Moreover, the data in EMRP is a valuable source of knowledge regarding motivation for and expansion of the Extract Method Refactoring techniques in the software engineering domain.

    Identifying Extract Method Refactoring Opportunities Based on Functional Relevance

    Summary: This paper conducts a study across open-source and industrial use cases taking raw data as input from repositories to apply agglomerative clustering (using Single Responsibility Principle) to form decision metrics on method size and Cohesion and then provides Extract Method Refactoring opportunities for Long Methods based on the set thresholds. The complete approach is know as SEMI.


    Orginal Paper Link: SEMI

    Data & Tools Link: Download Data Tools section.


    Data Details:

    • Industrial use case was a printer company dataset: Two Long Methods (408 LOC and 642 LOC).
    • Open-source use case were examined OSS projects: Wikidev, MyPlanner, MyWebMarket, Junit, and JHotDraw (132 methods with 155 identified extract method opportunities).

    Approach Overview:

    Step 1:

    • List out all the distinct variables, methods calls in a code.

    Step 2:

    • Create a table containing distinct variables/methods and their repetition in each statement.

    Step 3:

    • Use agglomerative hierarchical clustering to group the statements based on similarity of used words. (Increase steps to reduce clusters and maintain code syntax and semantics)

    Step 4:

    Form a list of all candidate Extract Method opportunities, based on decided clusters. (Example 2-32, 2-34 lines to be in two independent methods)

    • The primary measure of benefit: Cohesion — Benefit measurement occurs through the use of LCOM, a metric that assesses method cohesion and is correlated to the methods size or CC method which checks for cohesion.
    • A secondary measure of benefit: Method Size (Lines of code to be extracted)

    Step 5:

    Setting the Primary suggestion for extraction out of all groups of opportunities (clusters) is based on the use of cohesion evaluation metrics:

    • Max size difference (diff size is less than max diff size - threshold is 0.2)
    • Minimum Overlap  (overlap > minimum overlap — threshold is 0.1)
    • Difference b/w benefits 


    Step 6:

    Accuracy of approach for Long Methods is compared to other approaches such as JDeodrant and JExtract using the following Evaluation metrics:

    • Recall
    • Precision
    • F-measure

    Identifying Extract Method Refactorings

    Summary: This study makes use of open-source data to apply Segmentation to extract refactoring opportunities that are more limited and precise. The approach makes use of graphs to form relationships b/w statements using nodes and vertices. These relationships are clustered based on metrics thresholds to identify the best opportunities to refactor code.


    Orginal Paper Link: Segmentation

    Data & Tools Link: Download Data Tools section.


    Data Details:

    • Open-source projects were JUnit, JHotDraw, MyWebMarket, EventBus, Mockito, and Xdata.
    • This methodology applies extract method refactoring to any of the following; long method, code clone, or feature envy in the code.

    Approach Overview: Segmentation, a graph-based clustering approach applies the follow methodology to detect refactor opportunities.


    Step 1: Extract Method Opportunities Detection

    This approach makes uses of clone genealogy analysis to create clone labels based on software version history.


    • Two-step segmentation is conducted to detect EMO using the following criteria:
      • Block Association Clustering (BAC): Focuses on the extraction of statements with beginning blocks such as if, for, while try and switch. The entry statement is always clustered with all the statements of the block, including nest and non-secondary blocks such as else, if else, and catch, etc.
      • Define-use Association Clustering (DAC): Focuses on preventing extensive generation of large cluster counts by associating variable statements to the block association.


    Step 2: Identify Best Extract Method Opportunities

    Segmentation approach makes use of two metrics Lack of Computational Strength (LoCS) and Parent Affinity (PA) which are computed over a segment graph formed using vertices represent statements of method and edges capture association among statements: 


    • Lack of Computational Strength (LoCS) aims at classifying a cluster of statements as a core-functional block. It analyses data sharing among statements within and outside the cluster.
    • Parrent Affinity (PA) utilizes the data sharing present b/w two consecutive blocks to determine if both the blocks contribute to the same functionality, in a bottom-up approach.


      • Once the cluster is marked as EMO, all statements are executed for any other clustering consideration, making the EMO disjoint, and lowering the total suggestions. Values below the default threshold for the LoCS metric is 0.41, and for PA it is 0.34 will yield in an EMO, clusters with a higher LoCS will not be considered for extraction for refactoring.


      Step 3:

      Accuracy of approach for Long Methods is compared to other approaches such as SEMI and JExtract using the following Evaluation metrics:

      • Recall
      • Precision
      • F-measure


    Automatic Clone recommendation for Refactoring based on the present and the past

    Summary: This study uses open-source repos to recommend refactoring of cloned repositories using CREC approach. The CREC approach labels repos as cloned or non-cloned based on genelogy analysis. These label repo codes are then utilize to review for refactoring. Features are formed based on the labeled clone refactor and non factored repositories. These features are utilized to create a predictive model for providing recommendations.


    Orginal Paper Link: CREC

    Data & Tools Link: Download Data Tools section.


    Data Details:

    • Open-source projects used for this approach are Axis2, Eclipse.jdt.core, Elastic Search, JFreeChart, JRuby, and Lucene.

    Approach Overview: CREC, a learning-based approach applies the following methodology to detect clones, and then identify the cloned repositories for refactoring.


    Step 1: Detection of Clone and Non-Clone Repositories

    This approach makes uses of clone genealogy analysis to create clone labels based on software version history.


    • Two-step segmentation is conducted to detect EMO using the following criteria:
      • Multi-Version Clone Detection: SourcererCC tool utilized to find block-level clones with at least 6 Loc or 30 tokens (set mini similarity threshold to 80%). CREC samples versions through chronological order and then based on the amount of code revisions  b/w each version (e.g., 200 b/w S1 and S2, S2, and S3, for the same clone repo).
      • Cross-Version Clone Detection: Utilize an existing approach by “X. Wang, Y.Dang” to match clones across versions based on code location and textual similarity (e.g., A and A’ in the same file are more similar than A and B in the same file).


    Step 2: Detection of Refactoring Clone and Non-Refactoring Clone Repositories

    This step uses heuristics to decide whether a clone group was refactored or not, with the following criteria — (Manual work is required for labeling)


    • The latest 2 clone instances in the group have code size reduced when evolving to the next versions.
    • In the above subset from 1, at least 2 clone instances add invocations to a method m() during the evolution.
    • In the above subset from 2, at least 2 clone members have their removed code like m()’s body. (Similarity threshold at 0.4)

    Step 3: Feature Engineering

    The author forms 34 features from various code aspects to reflect on the history and present status of the program.

    • Clone Code Features: LOC, complexity, independence — 11 features
    • Clone History Features: file existence, dir. & file changes, the latest 1/10 of the project’s sampled commits — 6 features
    • Relative Location Features among Clones : clones located in the same package, files, and classes — 6 features
    • Syntactic Difference Features among Clones:: divergent identifier usage of variables, methods, and types using the MCIDIff tool. — 6 features
    • Co-Change Features among Clones: to reflect the co-change relationship among clones where refactoring is conducted multiple times — 5 features


    Step 4: Training and Testing

    • Used Ada boost algorithm on Weka with 50 weak learner iterations on 34 features to train the model.
    • Groups predicted to be refactored with over 50% likelihood recommended for refactoring, if multiple groups are recommended then, CREC ranks them based on likelihood outputs.

    Step 5: Evaluation

    Accuracy of feature set formed by CREC approach is compared to Wang et al.'s approach using evaluation metrics:

    • Recall
    • Precision
    • F-measure


    Just-in-time code duplicates extraction

    Summary: This paper uses open-source repos to form a recommend system for detection and refactoring of code duplicates on the fly known as Anticopypaster, using Extract method IDE capability.


    Orginal Paper Link: Anticopypaster

    Data & Tools Link: Download Data Tools section.


    Data Details:

    • 13 Open-source Apache projects are reviewed to cumulate 18942 code fragments.

    Approach Overview:

    Step 1: Detection of refactoring extract methods

    • Refactoring Miner v2.0 is used to detect 9,471 positive cases of Extract Method refactoring.
    • A ranking formula inspired by the work of Haas and Hummel was used to discover negative samples, by selecting the sample at the bottom of the list; code fragments that are less likely to be extracted. Using three independent code metrics: statement length, nesting depth, and nesting, the fragments were ranked and selection was conducted.


    Step 2: Feature Engineering

    The author formed 78 metrics utilizing previous studies.

    • Metrics that relate to the code fragment: e.g., length of the code fragment in symbols, keyword count, etc.
    • Metrics that relate to the enclosing method: e.g., e.g., length of the enclosing method in lines, etc.
    • Code metrics e.g., number of references to fields from the enclosing class in the code fragment, etc.

    Step 3: Training

    • Detection of an Extract Method opportunity as a binary classification problem.
    • Convolutional Neural Network (CNN), which recommends appropriate Extract Method refactoring with an F-measure of 0.82.


    Step 4: Tool for inference

    The author formed 78 metrics utilizing previous studies.

    • Duplicate Detector:A code similarity-based approach detects duplicates by parsing code fragments into tokens and comparing their similarity, considering minor changes. Posted code is delayed to prevent changes, checked for validity if the code is Java using PSI representation, and it is sent to the Code Analyzer if duplicates persist.
    • Code Analyzer: makes use of the duplicated fragment and uses the PSI representation from 78 metrics.
    • Method Extractor The input vector of metrics for the fragment is fed to the pre-trained model to identify its similarity to existing refactored fragments used in training, if successful then the refactoring launcher is called.
    • Refactoring Launcher checks if the fragment can be separated to another method, if checks clear, a notification to the user is issued. On user acceptance, the IDE Extract method API is called to proceed.



    Step 5: Evaluation

    Accuracy of approach using CNN is compared to other machine learning models and the following evaluation metrics are used:

    • Recall
    • Precision
    • F-measure
    • PR-AUC
    • McNemar's test


    JExtract: An Exclipse plug-in for recommending Automated Extract Method Refactorings

    Summary: JExtract implements a novel approach for recommending automated Extract Method refactorings. The tool was designed as a plug-in for the Eclipse IDE that automatically identifies, ranks, and applies the refactoring.


    Orginal Paper Link: JExtract

    Data & Tools Link: Download Data Tools section.


    Data Details:

    • Open-source projects used for this approach are JHotDraw 5.2, JUnit 3.8, and MyWebMarket.



    Approach Overview:

    Step 1: Detection of refactoring extract methods

    JExtract makes use of an algorithm to extract statements for Extract method refactoring. The statements are first divided into blocks. Candidates can only be generated from individual blocks; Candidates cannot be extracted from S2.6 & S3.1 at the same time. When a statement is selected, the respective child statements are also included e.g., S3.2 to S3.7.


    Some other base conditions are:

    • A candidate recommendation must respect a size threshold defined by the parameter Minimum Extracted statements. (Have at least 3 statements)
    • a candidate recommendation must respect the preconditions defined by the Extract Method refactoring engine.


    Step 2: Scoring

    The software makes use of Move Method recommendation heuristics, by minimizing the structural similarity b/w the selected and remaining statements. For instance, assume m’ as the highlighted code in above figure (i.e., an Extract Method candidate) and m’’ the remaining statements in the original method mouseReleased. Set of m’ = (metaType, fc, fcbg). On the other hand, the set of m’’ = (metaType, btn,  cy,  cx,  cw,  ch, buttons, me, rect). In this case, the intersection between these two sets contains only metaType, which shows m’ to be cohesive and decoupled from m’’. The similarity is calculated using Kulczynski similarity coefficient.



    Step 3: Tool for inference

    The third step involves sorting and filtering of candidates according to the following parameters:

    • Maximum Recommendations per Method - 3 for each method.
    • Minimum Score Value - This has to be configured when the user desires to set a minimum dissimilarity threshold.

    Step 4: Evaluation

    Accuracy of JExtract over Top-k Recommendations per Method is compared to JDeodorant over three datasets using the following evaluation metrics:

    • Recall
    • Precision

    Adventure of a lifetime: Extract Method Refactoring for Rust

    Summary: The author of this paper designs and implements an automated Extract Method refactoring tool (REM) for the Rust programs, considering the challenges imposed by Rust’s ownership and lifetime-based type system.


    Orginal Paper Link: REM

    Data & Tools Link: Download Data Tools section.


    Data Details:

    The research utilized a corpus of changes (40 experiments) in five popular Rust projects for evaluation, namely:

    • petgraph; a graph library.
    • gitoxide; a Rust implementation of git.
    • kickoff, a GUI program launcher.
    • sniffnet, a network packet sniffer.
    • beerus, a lightweight client for the Starknet decentralized network.

    Approach Overview:

    Step 1: Systematic Decomposition

    THe paper broke down the Extract Method refactoring process for Rust programs into several distinct stages, aiming to align with Rust's type safety requirements. This systematic approach allowed them to develop a clear understanding of the unique challenges posed by Rust's ownership and lifetime-based type system. By decomposing the problem, they were able to identify specific areas where traditional refactoring techniques might not suffice.


    Step 2: Formulation of Extract Method

    Extract Method was formulated as a multi-step process, beginning with the initial extraction of the code segment and followed by a series of repair procedures. This included both naïve function hoisting and subsequent sophisticated repair procedures, which were necessary to ensure that the resulting program complied with Rust's strict type system. The formulation also took into account the need for precise ownership annotations and appropriate handling of lifetime constraints, ensuring that the refactored code remained well-typed.



    Step 3: Automated Analysis and Repair Procedures

    The solution involved a series of automated analyses and repair procedures, including:

    • A novel static intra-procedural ownership analysis for inferring precise ownership annotations for the extracted function's parameters. The key features of the analysis were:
      • Ordered Abstract Domain for Ownership Annotations
      • Inference of Ownership Annotations
      • Consideration of Borrowing and Mutation
      • Handling Data Flow and Aliases
    • Utilization of the Rust compiler's lifetime checker for the appropriate annotation of lifetime constraints in the extracted function's signature.


    Step 4: Tool Implementation (REM)

    The culmination of the research efforts led to the development of a tool named REM, which served as an automated Extract Method refactoring solution integrated with the IntelliJ IDEA plugin for Rust. This tool was designed with a focus on expanding the capabilities of traditional refactoring tools to accommodate the complexities of Rust's type system. REM's implementation facilitated the extraction of a wider range of feature-rich code fragments into semantically correct functions. It was also capable of replicating manual method extractions performed by human developers, thus highlighting its effectiveness and reliability in practical development scenarios.
